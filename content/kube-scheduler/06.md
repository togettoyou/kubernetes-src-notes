前面几回，我们分别介绍了调度队列、调度缓存和事件处理器。这三个模块各司其职，共同支撑着调度器的运转。其中，调度队列的实现已经提前接触到了调度框架的两个扩展点：PreEnqueue 和 QueueSort。本文将深入分析调度框架本身——它是如何定义扩展点、如何管理插件、如何在调度流程中逐一调用这些插件的。

调度框架的核心代码位于 `pkg/scheduler/framework` 目录下，结构如下：

```bash
$ tree pkg/scheduler/framework/ --max-depth=1

├── cycle_state.go   # 调度周期内的共享状态
├── interface.go     # 所有扩展点的接口定义

    ├── registry.go          # 插件注册表
    └── waiting_pods_map.go  # Permit 等待 Pod 的管理
```

### CycleState：调度周期内的共享状态

在一次完整的调度流程中，需要经过多个阶段，每个阶段可能有多个插件依次执行。很自然地会产生一个需求：前面插件的计算结果，能不能被后面的插件复用，而不是重复计算？

这就是 `CycleState` 的作用。它是一个贯穿整个调度周期的"黑板"，每个插件都可以向其中写入或读取数据，以实现跨插件的信息共享：

```go
// CycleState 为插件提供了一种在调度周期内存储和读取任意数据的机制。
// 一个插件写入的数据，可以被另一个插件读取、修改或删除。
// 注意：CycleState 不提供任何数据保护，因为所有插件都被视为可信的。
// 底层使用 sync.Map，针对"写一次、读多次"的模式进行了优化。
// 推荐模式：在 PreFilter/PreScore 阶段写入，在 Filter/Score 阶段读取。
type CycleState struct {
    storage sync.Map
    // 是否记录插件执行耗时指标
    recordPluginMetrics bool
    // 在 PreFilter 阶段返回 Skip 的插件集合，对应的 Filter 插件将被跳过
    SkipFilterPlugins sets.Set[string]
    // 在 PreScore 阶段返回 Skip 的插件集合，对应的 Score 插件将被跳过
    SkipScorePlugins sets.Set[string]
}

// StateKey 是存储在 CycleState 中的键类型
type StateKey string

// StateData 是存储在 CycleState 中的值的接口，需要实现 Clone 方法
type StateData interface {
    Clone() StateData
}

// Read 根据 key 从 CycleState 中读取数据，key 不存在时返回错误
func (c *CycleState) Read(key StateKey) (StateData, error) {
    if v, ok := c.storage.Load(key); ok {
        return v.(StateData), nil
    }
    return nil, ErrNotFound
}

// Write 将数据写入 CycleState
func (c *CycleState) Write(key StateKey, val StateData) {
    c.storage.Store(key, val)
}

// Delete 从 CycleState 中删除指定 key 的数据
func (c *CycleState) Delete(key StateKey) {
    c.storage.Delete(key)
}

// Clone 创建一个 CycleState 的副本，在抢占等场景下需要 Clone 一个 CycleState 进行模拟调度
func (c *CycleState) Clone() *CycleState {
    if c == nil {
        return nil
    }
    copy := NewCycleState()
    c.storage.Range(func(k, v interface{}) bool {
        copy.storage.Store(k, v.(StateData).Clone())
        return true
    })
    copy.recordPluginMetrics = c.recordPluginMetrics
    copy.SkipFilterPlugins = c.SkipFilterPlugins
    copy.SkipScorePlugins = c.SkipScorePlugins
    return copy
}
```

举一个具体例子，`InterPodAffinity` 插件在 `PreFilter` 阶段会计算当前集群中所有 Pod 的亲和性规则并将结果写入 `CycleState`，之后 `Filter` 阶段就可以直接从 `CycleState` 读取，而不用重复计算。

### 扩展点接口

`interface.go` 文件是整个调度框架的"规范说明书"，它为每个扩展点定义了对应的 Go 接口，所有调度插件都必须实现其中至少一个。

所有插件都需要实现最基础的 `Plugin` 接口，用于提供插件名称：

```go
type Plugin interface {
    Name() string
}
```

以下是全部扩展点接口的定义与说明：

**PreEnqueue（入队前）**

```go
// PreEnqueuePlugin 在 Pod 加入 activeQ 前被调用，用于控制哪些 Pod 允许入队。
// 实现该接口的插件应轻量高效，避免耗时操作，否则会阻塞其他 Pod 入队。
type PreEnqueuePlugin interface {
    Plugin
    PreEnqueue(ctx context.Context, p *v1.Pod) *Status
}
```

**QueueSort（队列排序）**

```go
// QueueSortPlugin 用于对调度队列中的 Pod 进行排序，决定哪个 Pod 优先被调度。
// 同一时刻只能启用一个 QueueSort 插件。
type QueueSortPlugin interface {
    Plugin
    Less(*QueuedPodInfo, *QueuedPodInfo) bool
}
```

**PreFilter（过滤前）**

```go
// PreFilterPlugin 在调度周期开始时被调用，用于预处理 Pod 的调度信息。
// 可选地返回一个 PreFilterResult，用于缩小下游需要评估的节点范围。
// 返回 Skip 时，对应的 Filter 插件和 PreFilterExtensions 将在本轮调度中被跳过。
type PreFilterPlugin interface {
    Plugin
    PreFilter(ctx context.Context, state *CycleState, p *v1.Pod) (*PreFilterResult, *Status)
    // 可选：返回增量更新的扩展接口（AddPod/RemovePod），在抢占场景中使用。
    PreFilterExtensions() PreFilterExtensions
}

// PreFilterResult 用于缩小节点范围，如果 NodeNames 为 nil，则所有节点都参与后续评估。
type PreFilterResult struct {
    NodeNames sets.String
}
```

**Filter（过滤）**

```go
// FilterPlugin 用于筛选不满足 Pod 调度条件的节点，原概念叫"predicate"（预选）。
// 返回 Success 表示节点可以运行该 Pod，否则返回 Unschedulable 或 Error。
type FilterPlugin interface {
    Plugin
    Filter(ctx context.Context, state *CycleState, pod *v1.Pod, nodeInfo *NodeInfo) *Status
}
```

**PostFilter（过滤后）**

```go
// PostFilterPlugin 在所有节点都无法通过 Filter 时被调用，默认实现是抢占（Preemption）。
// 返回 Success 表示找到了使 Pod 可调度的方法，返回 Unschedulable 表示仍然无法调度。
type PostFilterPlugin interface {
    Plugin
    PostFilter(ctx context.Context, state *CycleState, pod *v1.Pod, filteredNodeStatusMap NodeToStatusMap) (*PostFilterResult, *Status)
}
```

**PreScore（打分前）**

```go
// PreScorePlugin 在过滤阶段结束后，对通过过滤的节点列表进行预处理。
// 它是一个信息性扩展点，常用于更新 CycleState，供 Score 插件使用。
// 返回 Skip 时，对应的 Score 插件将在本轮调度中被跳过。
type PreScorePlugin interface {
    Plugin
    PreScore(ctx context.Context, state *CycleState, pod *v1.Pod, nodes []*v1.Node) *Status
}
```

**Score（打分）**

```go
// ScorePlugin 对通过过滤的节点进行打分，原概念叫"priority"（优选）。
// Score 方法对每个节点单独打分，NormalizeScore 方法将分数归一化到 [0, 100] 范围内。
type ScorePlugin interface {
    Plugin
    Score(ctx context.Context, state *CycleState, p *v1.Pod, nodeName string) (int64, *Status)
    // ScoreExtensions 返回归一化扩展，如果不需要归一化则返回 nil。
    ScoreExtensions() ScoreExtensions
}

type ScoreExtensions interface {
    NormalizeScore(ctx context.Context, state *CycleState, p *v1.Pod, scores NodeScoreList) *Status
}
```

**Reserve（预留）**

```go
// ReservePlugin 用于在调度缓存中"预留"资源。
// Reserve 在节点选定后、绑定前调用，成功后才进入绑定周期。
// 如果 Reserve 失败，调用方负责调用 Unreserve 回滚已预留的资源。
// Unreserve 必须是幂等的，即使对应的 Reserve 没有被调用也要能安全执行。
type ReservePlugin interface {
    Plugin
    Reserve(ctx context.Context, state *CycleState, p *v1.Pod, nodeName string) *Status
    Unreserve(ctx context.Context, state *CycleState, p *v1.Pod, nodeName string)
}
```

**Permit（许可）**

```go
// PermitPlugin 在 PreBind 之前被调用，可以阻止或延迟 Pod 的绑定。
// 返回 Success 表示允许绑定，返回 Wait 表示等待（可设置超时时间），返回 Unschedulable 表示拒绝。
// Permit 最大等待时间为 15 分钟。
type PermitPlugin interface {
    Plugin
    Permit(ctx context.Context, state *CycleState, p *v1.Pod, nodeName string) (*Status, time.Duration)
}
```

**PreBind（绑定前）**

```go
// PreBindPlugin 在正式绑定之前执行，用于完成绑定所需的前置工作。
// 例如，volumebinding 插件会在此阶段完成 PVC 的绑定。
type PreBindPlugin interface {
    Plugin
    PreBind(ctx context.Context, state *CycleState, p *v1.Pod, nodeName string) *Status
}
```

**Bind（绑定）**

```go
// BindPlugin 负责将 Pod 绑定到节点。
// 每个 Bind 插件按顺序被调用，可以选择跳过（返回 Skip）或处理（返回 Success/Error）。
// 一旦有插件成功处理，后续的 Bind 插件将被跳过。
type BindPlugin interface {
    Plugin
    Bind(ctx context.Context, state *CycleState, p *v1.Pod, nodeName string) *Status
}
```

**PostBind（绑定后）**

```go
// PostBindPlugin 在 Pod 成功绑定后被调用，为通知性扩展点，常用于清理工作。
type PostBindPlugin interface {
    Plugin
    PostBind(ctx context.Context, state *CycleState, p *v1.Pod, nodeName string)
}
```

### Status：插件的返回状态

所有扩展点的运行结果都通过 `*Status` 来传递：

```go
type Status struct {
    code         Code    // 状态码
    reasons      []string // 原因说明
    err          error   // 内部错误
    failedPlugin string  // 失败的插件名称
}

// 预定义的状态码
const (
    Success                      Code = iota // 成功（nil Status 也视为 Success）
    Error                                    // 内部错误
    Unschedulable                            // 当前节点不可调度，可通过 PostFilter 补救
    UnschedulableAndUnresolvable             // 完全不可调度，PostFilter 也无法解决
    Wait                                     // Permit 插件要求等待
    Skip                                     // 插件选择跳过当前处理
)
```

### 插件注册表

在分析 `frameworkImpl` 之前，先了解插件是如何注册和管理的。注册表 `Registry` 是一个 `map`，将插件名称映射到对应的工厂函数：

```go
// PluginFactory 是一个插件工厂函数，接收配置参数和框架句柄，返回一个插件实例。
type PluginFactory = func(configuration runtime.Object, f framework.Handle) (framework.Plugin, error)

// Registry 是所有可用插件的集合。框架通过 Registry 来启用和初始化插件。
// 所有插件必须在框架初始化之前先注册到 Registry 中。
type Registry map[string]PluginFactory

// Register 向注册表中添加新插件，同名插件已存在时返回错误。
func (r Registry) Register(name string, factory PluginFactory) error {
    if _, ok := r[name]; ok {
        return fmt.Errorf("a plugin named %v already exists", name)
    }
    r[name] = factory
    return nil
}

// Merge 将另一个注册表合并到当前注册表，用于合并 InTree 和 OutOfTree 插件。
func (r Registry) Merge(in Registry) error {
    for name, factory := range in {
        if err := r.Register(name, factory); err != nil {
            return err
        }
    }
    return nil
}
```

工厂函数 `PluginFactory` 的设计非常精妙：它接收 `framework.Handle` 作为参数，插件通过 `Handle` 可以获取到 clientSet、SharedInformerFactory、缓存快照等调度器内部资源，从而实现与调度器的解耦。以 NodeAffinity 插件为例：

```go
// NodeAffinity 插件实现了 PreFilter、Filter、PreScore、Score 多个扩展点。
func New(plArgs runtime.Object, h framework.Handle) (framework.Plugin, error) {
    args, err := getArgs(plArgs)
    if err != nil {
        return nil, err
    }
    pl := &NodeAffinity{
        handle: h, // 保存 handle，后续调度逻辑中使用
    }
    // ...
    return pl, nil
}
```

### 调度框架的初始化

`NewFramework` 函数负责根据配置和注册表初始化调度框架，核心逻辑如下：

```go
func NewFramework(r Registry, profile *config.KubeSchedulerProfile, stopCh <-chan struct{}, opts ...Option) (framework.Framework, error) {
    // 应用可选配置（clientSet、informerFactory、parallelism 等）
    options := defaultFrameworkOptions(stopCh)
    for _, opt := range opts {
        opt(&options)
    }

    f := &frameworkImpl{
        registry:             r,
        snapshotSharedLister: options.snapshotSharedLister,
        scorePluginWeight:    make(map[string]int),
        waitingPods:          newWaitingPodsMap(),
        // ......
    }

    // 根据 profile 中的配置，确定需要实例化哪些插件
    pg := f.pluginsNeeded(profile.Plugins)

    pluginsMap := make(map[string]framework.Plugin)
    for name, factory := range r {
        // 只初始化需要的插件
        if !pg.Has(name) {
            continue
        }
        // 通过工厂函数实例化插件，传入配置参数和 Handle（即 frameworkImpl 自身）
        p, err := factory(args, f)
        pluginsMap[name] = p
        // 顺便更新 clusterEventMap，记录插件关心哪些集群事件（见调度队列章节）
        fillEventToPluginMap(p, options.clusterEventMap)
    }

    // 将插件按扩展点分组，填充 f.preFilterPlugins, f.filterPlugins 等列表
    for _, e := range f.getExtensionPoints(profile.Plugins) {
        if err := updatePluginList(e.slicePtr, *e.plugins, pluginsMap); err != nil {
            return nil, err
        }
    }

    // 约束：QueueSort 插件只能有一个，Bind 插件至少需要一个
    if len(f.queueSortPlugins) != 1 {
        return nil, fmt.Errorf("only one queue sort plugin required ...")
    }
    if len(f.bindPlugins) == 0 {
        return nil, fmt.Errorf("at least one bind plugin is needed ...")
    }

    return f, nil
}
```

两个 `for` 循环结合来看：第一个循环对每个插件只调用一次工厂函数，实例存入 `pluginsMap`；第二个循环遍历所有扩展点，从同一个 `pluginsMap` 按名字取出实例追加到对应切片。正因如此，一个实现了多个扩展点接口的插件（如 `NodeAffinity` 同时实现了 PreFilter、Filter、PreScore、Score），会同时出现在 `preFilterPlugins`、`filterPlugins`、`scorePlugins` 等多个列表中，但它只被实例化了一次。

`updatePluginList` 方法利用反射，将插件实例按类型分配到对应的扩展点切片中：

```go
func updatePluginList(pluginList interface{}, pluginSet config.PluginSet, pluginsMap map[string]framework.Plugin) error {
    plugins := reflect.ValueOf(pluginList).Elem() // 获取切片本身（如 []framework.PreFilterPlugin）
    pluginType := plugins.Type().Elem()           // 获取元素类型（如 PreFilterPlugin）

    for _, ep := range pluginSet.Enabled {
        pg, ok := pluginsMap[ep.Name]
        // 用反射检查插件是否实现了该扩展点的接口
        if !reflect.TypeOf(pg).Implements(pluginType) {
            return fmt.Errorf("plugin %q does not extend %s plugin", ep.Name, pluginType.Name())
        }
        // 追加到切片
        newPlugins := reflect.Append(plugins, reflect.ValueOf(pg))
        plugins.Set(newPlugins)
    }
    return nil
}
```

### 插件的调用：Run* 方法

框架初始化完成后，就可以在调度的各个阶段调用对应的 `Run*` 方法来执行插件了。

#### RunPreFilterPlugins

```go
func (f *frameworkImpl) RunPreFilterPlugins(ctx context.Context, state *framework.CycleState, pod *v1.Pod) (_ *framework.PreFilterResult, status *framework.Status) {
    var result *framework.PreFilterResult
    skipPlugins := sets.New[string]()

    for _, pl := range f.preFilterPlugins {
        r, s := f.runPreFilterPlugin(ctx, pl, state, pod)
        if s.IsSkip() {
            // 插件返回 Skip，将该插件名加入 skipPlugins，
            // 对应的 Filter 插件将在本轮调度中被跳过
            skipPlugins.Insert(pl.Name())
            continue
        }
        if !s.IsSuccess() {
            // 任何失败都会中断调度周期
            return nil, s
        }
        // 多个插件返回的 PreFilterResult 取交集，
        // 确保只有所有插件都认可的节点才参与后续过滤
        result = result.Merge(r)
        if !result.AllNodes() && len(result.NodeNames) == 0 {
            return nil, framework.NewStatus(framework.Unschedulable, ...)
        }
    }
    // 将需要跳过的 Filter 插件名集合写入 CycleState
    state.SkipFilterPlugins = skipPlugins
    return result, nil
}
```

PreFilter 阶段的设计体现了两个重要功能：

1. **通过 CycleState 预计算**：在昂贵的 Filter 开始前，集中完成可以复用的计算（如亲和性规则解析），供后续插件读取。
2. **收窄候选节点集合**：通过返回 `PreFilterResult`，插件可以直接圈定候选节点，缩短后续 Filter 的遍历范围（如 `NodeAffinity` 插件在 PreFilter 阶段根据 Pod 的亲和性规则计算出满足条件的节点集合，后续 Filter 只需对这部分节点进行评估）。

#### RunFilterPlugins

```go
func (f *frameworkImpl) RunFilterPlugins(
    ctx context.Context,
    state *framework.CycleState,
    pod *v1.Pod,
    nodeInfo *framework.NodeInfo,
) *framework.Status {
    for _, pl := range f.filterPlugins {
        // 跳过在 PreFilter 阶段被标记为 Skip 的插件
        if state.SkipFilterPlugins.Has(pl.Name()) {
            continue
        }
        if status := f.runFilterPlugin(ctx, pl, state, pod, nodeInfo); !status.IsSuccess() {
            // 任何插件拒绝，该节点就被排除
            status.SetFailedPlugin(pl.Name())
            return status
        }
    }
    return nil
}
```

Filter 是对单个节点执行的，调度器会并行地对所有候选节点调用 `RunFilterPlugins`，只要有一个插件返回非 Success，该节点就会被排除。

#### RunScorePlugins

```go
func (f *frameworkImpl) RunScorePlugins(ctx context.Context, state *framework.CycleState, pod *v1.Pod, nodes []*v1.Node) (ns []framework.NodePluginScores, status *framework.Status) {
    allNodePluginScores := make([]framework.NodePluginScores, len(nodes))
    numPlugins := len(f.scorePlugins) - state.SkipScorePlugins.Len()
    // 过滤掉在 PreScore 阶段被标记为 Skip 的插件，同时初始化每个插件的节点打分列表
    plugins := make([]framework.ScorePlugin, 0, numPlugins)
    pluginToNodeScores := make(map[string]framework.NodeScoreList, numPlugins)
    for _, pl := range f.scorePlugins {
        if state.SkipScorePlugins.Has(pl.Name()) {
            continue
        }
        plugins = append(plugins, pl)
        pluginToNodeScores[pl.Name()] = make(framework.NodeScoreList, len(nodes))
    }

    // 第一步：并行对所有节点调用每个 Score 插件的 Score 方法
    f.Parallelizer().Until(ctx, len(nodes), func(index int) {
        nodeName := nodes[index].Name
        for _, pl := range plugins {
            s, status := f.runScorePlugin(ctx, pl, state, pod, nodeName)
            pluginToNodeScores[pl.Name()][index] = framework.NodeScore{Name: nodeName, Score: s}
        }
    }, metrics.Score)

    // 第二步：并行对每个 Score 插件执行 NormalizeScore（归一化到 [0, 100]）
    f.Parallelizer().Until(ctx, len(plugins), func(index int) {
        pl := plugins[index]
        if pl.ScoreExtensions() == nil {
            return
        }
        nodeScoreList := pluginToNodeScores[pl.Name()]
        f.runScoreExtension(ctx, pl, state, pod, nodeScoreList)
    }, metrics.Score)

    // 第三步：并行对每个节点，计算所有插件的加权总分
    f.Parallelizer().Until(ctx, len(nodes), func(index int) {
        nodePluginScores := framework.NodePluginScores{
            Name:   nodes[index].Name,
            Scores: make([]framework.PluginScore, len(plugins)),
        }
        for i, pl := range plugins {
            weight := f.scorePluginWeight[pl.Name()]
            score := pluginToNodeScores[pl.Name()][index].Score
            weightedScore := score * int64(weight)
            nodePluginScores.Scores[i] = framework.PluginScore{Name: pl.Name(), Score: weightedScore}
            nodePluginScores.TotalScore += weightedScore
        }
        allNodePluginScores[index] = nodePluginScores
    }, metrics.Score)

    return allNodePluginScores, nil
}
```

Score 阶段有三个并行化步骤，框架内置了并行器 `Parallelizer`，默认并发度为 16：

- **第一步（按节点并行）**：对每个节点并行调用所有 Score 插件的 `Score` 方法，结果写入 `pluginToNodeScores[插件名][节点下标]`。
- **第二步（按插件并行）**：对每个插件并行执行 `NormalizeScore`，将该插件对所有节点的原始分数归一化到 `[0, 100]`。只有实现了 `ScoreExtensions()` 的插件才会执行此步骤，返回 `nil` 的插件直接跳过。
- **第三步（按节点并行）**：对每个节点并行计算加权总分。这一步会校验每个插件的归一化分数是否在 `[0, 100]` 范围内，越界则直接报错终止。最终每个节点的 `TotalScore` 是所有插件加权分数之和。

`RunScorePlugins` 返回 `allNodePluginScores` 后，由调用方根据 `TotalScore` 选出得分最高的节点，这部分逻辑属于调度流程，留到下一回介绍。

#### RunReservePluginsReserve 与 RunReservePluginsUnreserve

```go
// Reserve：按顺序调用所有 Reserve 插件，任一失败则停止
func (f *frameworkImpl) RunReservePluginsReserve(ctx context.Context, state *framework.CycleState, pod *v1.Pod, nodeName string) *framework.Status {
    for _, pl := range f.reservePlugins {
        if status := f.runReservePluginReserve(ctx, pl, state, pod, nodeName); !status.IsSuccess() {
            return status // 失败时，调用方负责调用 Unreserve 回滚
        }
    }
    return nil
}

// Unreserve：以 Reserve 调用顺序的逆序执行，确保正确回滚
func (f *frameworkImpl) RunReservePluginsUnreserve(ctx context.Context, state *framework.CycleState, pod *v1.Pod, nodeName string) {
    // 注意：逆序执行，类似"栈的弹出"
    for i := len(f.reservePlugins) - 1; i >= 0; i-- {
        f.runReservePluginUnreserve(ctx, f.reservePlugins[i], state, pod, nodeName)
    }
}
```

两个方法的设计有几处值得关注：

**逆序回滚**：Unreserve 对所有插件逆序执行，而不是只回滚已成功的插件。假设插件 A、B、C 依次注册，B 的 Reserve 失败——框架停止 Reserve 调用后，Unreserve 会按 C → B → A 的顺序全部执行一遍。这正是接口规范要求 Unreserve 必须幂等的原因：它可能被调用在从未执行过 Reserve 的插件上（如此例中的 C）。

**框架不自动触发 Unreserve**：Reserve 失败时，框架只返回错误，不会主动调用 Unreserve，由调用方（`schedule_one.go`）负责。实际上 Unreserve 有三个触发时机：Reserve 失败、Permit 拒绝、绑定阶段出错——三种情况的共同语义是"Pod 最终未能完成调度，需要撤销已预留的资源"。

#### RunPermitPlugins 与 WaitOnPermit

Permit 是调度框架中最独特的扩展点，它允许插件阻止或延迟 Pod 的绑定：

```go
func (f *frameworkImpl) RunPermitPlugins(ctx context.Context, state *framework.CycleState, pod *v1.Pod, nodeName string) *framework.Status {
    pluginsWaitTime := make(map[string]time.Duration)
    statusCode := framework.Success

    for _, pl := range f.permitPlugins {
        status, timeout := f.runPermitPlugin(ctx, pl, state, pod, nodeName)
        if !status.IsSuccess() {
            if status.IsUnschedulable() {
                // 插件直接拒绝，立即终止
                return status
            }
            if status.IsWait() {
                // 插件要求等待，记录等待时间（最大 15 分钟）
                if timeout > maxTimeout {
                    timeout = maxTimeout
                }
                pluginsWaitTime[pl.Name()] = timeout
                statusCode = framework.Wait
            }
        }
    }

    if statusCode == framework.Wait {
        // 创建一个 waitingPod 对象，加入等待 map 中
        waitingPod := newWaitingPod(pod, pluginsWaitTime)
        f.waitingPods.add(waitingPod)
        return framework.NewStatus(framework.Wait, ...)
    }
    return nil
}
```

理解 Permit 的关键，是搞清楚 `RunPermitPlugins` 和 `WaitOnPermit` 分别在哪里被调用：`RunPermitPlugins` 在**调度周期**（主 goroutine）里执行，`WaitOnPermit` 则在随后异步启动的**绑定周期**（独立 goroutine）里执行。两者之间通过 channel 传递信号。

具体流程如下：`RunPermitPlugins` 遍历所有 Permit 插件，若有插件返回 `Wait`，就把 Pod 包装成 `waitingPod` 存入 `waitingPods` map，然后直接返回 `Wait` 状态——调度周期结束，主 goroutine 立即去处理下一个 Pod。与此同时，绑定 goroutine 调用 `WaitOnPermit`，阻塞在 channel 上等待结果：

```go
// WaitOnPermit 会阻塞，直到 waitingPod 被 Allow 或 Reject
func (f *frameworkImpl) WaitOnPermit(ctx context.Context, pod *v1.Pod) *framework.Status {
    waitingPod := f.waitingPods.get(pod.UID)
    if waitingPod == nil {
        return nil // 没有等待，直接放行
    }
    defer f.waitingPods.remove(pod.UID)

    // 阻塞等待信号，直到某个外部调用 Allow 或 Reject
    s := <-waitingPod.s
    if !s.IsSuccess() {
        return s // 被拒绝
    }
    return nil // 被放行
}
```

那么谁来解除阻塞？是插件自己。Permit 插件可以通过 `Handle.GetWaitingPod(uid)` 取回对应的 `waitingPod`，在自己的异步逻辑（如另一个 goroutine 或回调）里调用 `Allow` 或 `Reject`，向 channel 发送信号，从而唤醒阻塞中的 `WaitOnPermit`。`Allow` 和 `Reject` 的实现如下：

```go
// waitingPod 存储了等待中的 Pod 和每个插件的超时计时器
type waitingPod struct {
    pod            *v1.Pod
    pendingPlugins map[string]*time.Timer // 每个插件对应一个超时计时器
    s              chan *framework.Status // 用于传递最终结果的 channel（缓冲大小为 1）
}

// Allow 表示该插件放行，若所有插件都放行则发送成功信号
func (w *waitingPod) Allow(pluginName string) {
    w.mu.Lock()
    defer w.mu.Unlock()
    // 停止该插件的超时计时器
    if timer, exist := w.pendingPlugins[pluginName]; exist {
        timer.Stop()
        delete(w.pendingPlugins, pluginName)
    }
    // 只有当所有插件都放行后，才向 channel 发送成功状态
    if len(w.pendingPlugins) != 0 {
        return
    }
    select {
    case w.s <- framework.NewStatus(framework.Success, ""):
    default:
    }
}

// Reject 表示该插件拒绝，立即停止所有计时器并发送失败状态
func (w *waitingPod) Reject(pluginName, msg string) {
    w.mu.RLock()
    defer w.mu.RUnlock()
    for _, timer := range w.pendingPlugins {
        timer.Stop()
    }
    select {
    case w.s <- framework.NewStatus(framework.Unschedulable, msg).WithFailedPlugin(pluginName):
    default:
    }
}
```

```go
func newWaitingPod(pod *v1.Pod, pluginsMaxWaitTime map[string]time.Duration) *waitingPod {
    wp := &waitingPod{
        pod: pod,
        s:   make(chan *framework.Status, 1),
    }
    wp.pendingPlugins = make(map[string]*time.Timer, len(pluginsMaxWaitTime))
    for plugin, waitTime := range pluginsMaxWaitTime {
        p, wt := plugin, waitTime
        // 超时后自动 Reject，确保 Pod 不会无限期等待
        wp.pendingPlugins[p] = time.AfterFunc(wt, func() {
            msg := fmt.Sprintf("rejected due to timeout after waiting %v at plugin %v", wt, p)
            wp.Reject(p, msg)
        })
    }
    return wp
}
```

还有两个细节值得关注：

- **多个插件都返回 Wait 时**：`pendingPlugins` 记录了所有还未放行的插件。每次 `Allow` 调用都会将对应插件从 `pendingPlugins` 中删除，只有当 `pendingPlugins` 变为空（即所有插件都放行）时，才向 channel 发送成功信号。任意一个插件调用 `Reject` 则立即发送失败信号。
- **超时保护**：每个等待中的插件都有一个 `time.AfterFunc` 计时器，超时后自动调用 `Reject`，避免 Pod 永久阻塞。最长等待时间上限为 15 分钟（`maxTimeout`）。

Permit 扩展点的典型应用场景是"Gang Scheduling"（组调度），即要求一批 Pod 要么全部调度成功，要么全部不调度。实现时，每个 Pod 到达 Permit 阶段后都返回 `Wait`，插件在自己的异步逻辑里持续检查组内所有 Pod 是否都已就绪，全部就绪后统一调用 `Allow` 放行，否则超时后调用 `Reject`。这类实现可以参考 [kubernetes-sigs/scheduler-plugins](https://github.com/kubernetes-sigs/scheduler-plugins/tree/master/pkg/coscheduling)。

#### RunBindPlugins

```go
func (f *frameworkImpl) RunBindPlugins(ctx context.Context, state *framework.CycleState, pod *v1.Pod, nodeName string) *framework.Status {
    for _, pl := range f.bindPlugins {
        status = f.runBindPlugin(ctx, pl, state, pod, nodeName)
        if status.IsSkip() {
            // 插件选择跳过，尝试下一个
            continue
        }
        if !status.IsSuccess() {
            return status // 绑定失败
        }
        return status // 绑定成功，后续 Bind 插件跳过
    }
    return status
}
```

Bind 扩展点的设计是"先到先得"：只要有一个插件处理了绑定（返回 Success 或 Error），后续插件就被跳过；如果插件选择不处理，则返回 Skip，让后续插件继续尝试。默认且唯一的内置绑定插件是 `DefaultBinder`，它通过调用 `clientSet.CoreV1().Pods(pod.Namespace).Bind(...)` 向 API Server 发送绑定请求。

### 总结

本文深入分析了 kube-scheduler 调度框架的实现：

- **CycleState** 是插件间共享状态的核心机制，基于 `sync.Map` 实现线程安全的读写，推荐在 PreFilter/PreScore 写入，在 Filter/Score 读取。
- **扩展点接口** 覆盖了 Pod 从入队（PreEnqueue）、过滤（PreFilter/Filter/PostFilter）、打分（PreScore/Score）到绑定（Reserve/Permit/PreBind/Bind/PostBind）的全生命周期。
- **插件注册表 Registry** 通过工厂函数模式解耦了插件实现与框架初始化，支持 InTree 和 OutOfTree 插件的合并。
- **frameworkImpl** 在初始化时根据配置实例化插件，并将其按扩展点分组存储，调度时通过 Run* 系列方法按序执行。
- **Permit 扩展点** 将调度周期与绑定周期分离：`RunPermitPlugins` 在调度周期（主 goroutine）执行，`WaitOnPermit` 在绑定周期（独立 goroutine）阻塞等待，两者通过 channel 传递信号，并配合超时计时器防止 Pod 永久阻塞，支持 Gang Scheduling 等高级调度场景。
